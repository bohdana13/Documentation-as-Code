Versioning Policy

Модель версійності

Для документації використовується схема MAJOR.MINOR.

  • 1.0 — базова узгоджена версія документації для стабільного релізу.
  • 1.1 — незначні уточнення або доповнення без зміни архітектурних рішень.
  • 2.0 — суттєва зміна вимог, архітектури або інфраструктури.

Правила актуалізації

  • При зміні функціональних вимог оновлюється SSD.
  • При зміні структури компонентів оновлюється SDD.
  • При зміні pipeline, середовища або розгортання оновлюється ISD.
  • Після цього оновлюються Test Strategy та Traceability Matrix.

Узгодження з релізами

  • Документація v1.0 відповідає релізу v1.0 системи.
  • Зміни API або сценаріїв користувача спричиняють підвищення версії документації.
