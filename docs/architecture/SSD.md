# SSD — System Specification Document

## 1. Опис системи
**EventMaster** — вебзастосунок для керування подіями, реєстрацією користувачів та бронюванням місць. Система орієнтована на організаторів подій, адміністраторів та кінцевих користувачів.

## 2. Зацікавлені сторони
| Роль | Опис потреб |
|---|---|
| Відвідувач | Перегляд подій, реєстрація, бронювання квитків |
| Організатор | Створення та редагування подій, керування лімітами місць |
| Адміністратор | Контроль користувачів, подій, категорій та звітності |

## 3. Функціональні вимоги
| ID | Вимога | Пріоритет |
|---|---|---|
| FR-01 | Система повинна дозволяти реєстрацію користувача | High |
| FR-02 | Система повинна дозволяти авторизацію користувача | High |
| FR-03 | Користувач повинен мати змогу переглядати список подій | High |
| FR-04 | Користувач повинен мати змогу переглядати деталі події | High |
| FR-05 | Користувач повинен мати змогу бронювати місце на подію | High |
| FR-06 | Організатор повинен мати змогу створювати події | High |
| FR-07 | Організатор повинен мати змогу редагувати власні події | Medium |
| FR-08 | Адміністратор повинен мати змогу видаляти події | Medium |
| FR-09 | Адміністратор повинен мати змогу керувати категоріями подій | Medium |
| FR-10 | Система повинна вести журнал змін і статусів бронювання | Medium |

## 4. Нефункціональні вимоги
| ID | Вимога |
|---|---|
| NFR-01 | Час відповіді API для типових запитів не повинен перевищувати 300 мс |
| NFR-02 | Інтерфейс повинен бути адаптивним для десктопа та мобільних пристроїв |
| NFR-03 | Аутентифікація повинна базуватися на JWT |
| NFR-04 | Паролі повинні зберігатися у вигляді хешу |
| NFR-05 | Документація повинна зберігатися в репозиторії у форматі Markdown |
| NFR-06 | Усі зміни документації повинні проходити через Git |

## 5. Бізнес-правила
- Один користувач не може забронювати більше місць, ніж доступно для події.
- Подія не може бути опублікована без назви, дати, місця та місткості.
- Лише автор події або адміністратор можуть її редагувати чи видаляти.

## 6. Обмеження
- Система реалізується як клієнт-серверний вебзастосунок.
- Основний формат документації — Markdown.
- База даних повинна підтримувати зберігання користувачів, подій, категорій та бронювань.
